# Fundara Multi-site Setup Guide

**Audience:** DevOps engineer / system administrator  
**Last updated:** 2026-06-19  
**Stack:** Ubuntu 24.04.4 LTS · Frappe Framework · ERPNext v16 · Fundara custom app  
**Decision reference:** D-06 (one site per org, workflow = template default)

---

## Overview

Fundara is deployed using the **one site per organisation** model (D-06, Opsi A). Each NGO or organisation that uses Fundara gets its own Frappe site with its own isolated database, isolated file storage, and its own workflow configuration. Multiple sites share the same application code and server infrastructure.

This guide covers:

1. How the multi-site architecture works
2. How to add a new organisation site from scratch
3. How to configure workflows per organisation (critical — read D-06 notes)
4. How to verify site isolation
5. How to decommission a site
6. Capacity planning

---

## 1. Architecture Overview

### How Frappe Bench Multi-site Works

Frappe Bench is designed to host multiple sites on a single server. The directory layout separates shared code from per-site data:

```
/home/frappe/frappe-bench/
├── apps/
│   ├── frappe/           ← shared code, same for all sites
│   ├── erpnext/          ← shared code, same for all sites
│   └── fundara/          ← shared code, same for all sites
│
├── sites/
│   ├── ngoa.fundara.id/  ← per-site: database, files, config
│   │   ├── site_config.json
│   │   ├── private/
│   │   └── public/
│   ├── ngob.fundara.id/
│   └── ngoc.fundara.id/
│
├── config/
│   └── nginx.conf        ← generated by bench, includes all sites
│
└── logs/
```

**What is shared across all sites:**
- Application code (Python, JavaScript, Jinja templates)
- Frappe workers and scheduler process (shared, but jobs are site-scoped)
- Redis cache and Redis queue instances
- Nginx process

**What is isolated per site:**
- MariaDB database (each site gets its own database)
- Private files (`sites/<sitename>/private/`)
- Public files (`sites/<sitename>/public/`)
- Site configuration (`sites/<sitename>/site_config.json`)
- Workflow configuration (Frappe Workflows live in the database, not in code)
- Users and roles
- All business data (funds, transactions, donors, reports)

### DNS Routing

Each organisation gets a subdomain:

```
ngoa.fundara.id   → Organisation A
ngob.fundara.id   → Organisation B
ngoc.fundara.id   → Organisation C
```

DNS configuration at the registrar/DNS provider: create an A record for each subdomain pointing to the server's public IP address.

```
ngoa.fundara.id.   IN A   <server-public-ip>
ngob.fundara.id.   IN A   <server-public-ip>
ngoc.fundara.id.   IN A   <server-public-ip>
```

Or use a wildcard A record if all subdomains of `fundara.id` resolve to the same server:

```
*.fundara.id.   IN A   <server-public-ip>
```

### Nginx Virtual Host per Site

Frappe generates an Nginx configuration automatically. When `bench setup nginx` is run, it creates a config that routes each hostname to the correct Frappe site. You do not write Nginx server blocks by hand for each site — Frappe handles this.

The generated config lives at:

```
/home/frappe/frappe-bench/config/nginx.conf
```

This file is symlinked (or included) in the system Nginx config:

```
/etc/nginx/conf.d/frappe.conf -> /home/frappe/frappe-bench/config/nginx.conf
```

After adding a new site, you must re-run `bench setup nginx` and reload Nginx.

## 2. Adding a New Organisation Site

Work through these steps in order. Each step must succeed before the next begins.

### Step 1: Gather Prerequisites

Before running any commands, confirm:

- [ ] The organisation's subdomain has been decided: `<orgname>.fundara.id`
- [ ] A DNS A record for this subdomain points to the server IP
- [ ] The organisation name, timezone, base currency, and default language are known
- [ ] An admin email address for the organisation has been provided
- [ ] The workflow configuration for this org has been discussed (see Section 3)

### Step 2: Create the Frappe Site

```bash
cd /home/frappe/frappe-bench

bench new-site <orgname>.fundara.id \
  --db-root-password <mariadb-root-password> \
  --admin-password <initial-admin-password>
```

This creates a new MariaDB database, initialises the Frappe schema, and creates the `site_config.json`.

Verify the site was created:

```bash
bench sites
# Should list <orgname>.fundara.id
ls sites/<orgname>.fundara.id/
# Should show: private/, public/, site_config.json
```

### Step 3: Install ERPNext on the Site

```bash
bench --site <orgname>.fundara.id install-app erpnext
```

This runs ERPNext migrations and loads ERPNext fixtures on the new site. It may take 3–5 minutes.

### Step 4: Install the Fundara App on the Site

```bash
bench --site <orgname>.fundara.id install-app fundara
```

This runs Fundara migrations and installs base Fundara DocTypes on the new site.

### Step 5: Configure Site-Specific Settings

Set the site defaults before running any wizard. These settings affect accounting and reporting.

```bash
bench --site <orgname>.fundara.id console
```

In the Frappe console, set the system settings:

```python
# Set timezone
frappe.db.set_default("time_zone", "Asia/Jakarta")  # adjust per org

# Set language
frappe.db.set_default("lang", "id")  # or "en" for English

# Commit and exit
frappe.db.commit()
exit()
```

Alternatively, these can be set through the UI after login: **Setup → System Settings**.

### Step 6: Run the ERPNext Setup Wizard

Log in to the site at `https://<orgname>.fundara.id` as Administrator with the password set in Step 2. The ERPNext setup wizard will appear on first login.

Complete the wizard with the organisation's details:
- **Company name:** the full legal name of the organisation
- **Country:** Indonesia (or the org's country)
- **Currency:** IDR (or the primary currency of the org — note D-04: multi-currency is enabled from the start)
- **Chart of Accounts:** select the NGO/nonprofit template if available, or the standard Indonesian template

After the wizard completes, the Company DocType is created and the Chart of Accounts is populated.

### Step 7: Run Fundara Setup Wizard (or Manual Setup)

Fundara provides a setup wizard or onboarding checklist to configure the Fundara-specific layer. The goal is to map ERPNext's Company/Cost Center structure to Fundara's Fund Stewardship and Organisation contexts.

**If a Fundara Setup Wizard is available:**

Navigate to **Fundara → Setup → Setup Wizard** and follow the prompts. The wizard will:
- Create the Organisation Profile
- Map Cost Centers to departments
- Import default workflow fixtures
- Create base Funding Source types (Grant, Donation, Campaign, Internal Reserve)
- Create placeholder Funds if the org provides initial fund information

**If setting up manually:**

```bash
# Import Fundara default fixtures
bench --site <orgname>.fundara.id import-fixtures --app fundara

# Import default Chart of Accounts extension for NGOs (if available as a fixture)
bench --site <orgname>.fundara.id import-fixtures --app fundara --doctype "Chart of Accounts"
```

Then in the browser, create:
1. **Organisation Profile** (Fundara → Organisation → Organisation Profile)
2. At least one **Funding Source** for each type of income the org receives
3. The initial **Fund** records if known at setup time

### Step 8: Configure DNS and Nginx

After creating the site, regenerate the Nginx configuration and reload:

```bash
bench setup nginx
sudo nginx -t                    # test config — must say "ok"
sudo systemctl reload nginx
```

Confirm the site is reachable over HTTP:

```bash
curl -s -o /dev/null -w "%{http_code}" http://<orgname>.fundara.id
# Should return 200 or 301 (redirect to HTTPS)
```

### Step 9: Set Up SSL Certificate

```bash
sudo certbot --nginx -d <orgname>.fundara.id --non-interactive --agree-tos \
  --email admin@fundara.id
```

Certbot will obtain a Let's Encrypt certificate and update the Nginx config automatically.

Verify HTTPS works:

```bash
curl -s -o /dev/null -w "%{http_code}" https://<orgname>.fundara.id
# Should return 200
```

Check certificate expiry is correctly scheduled for auto-renewal:

```bash
sudo certbot renew --dry-run
```

### Step 10: Import Default Workflow Templates

Fundara ships default workflow fixtures for the main financial documents. These are **templates** — they represent a reasonable default SOP, not a mandatory configuration. Every org will need to review and customise these.

Import the default workflow templates:

```bash
bench --site <orgname>.fundara.id import-fixtures --app fundara --doctype Workflow
```

This imports the following default workflows (as templates, not final configs):
- Cash Advance workflow
- Purchase Request workflow
- Fund Transfer workflow
- Cash Receipt workflow
- Liquidation workflow

After import, immediately proceed to Section 3 (Workflow Configuration per Organisation) to customise these templates to match the org's actual SOP before any users begin data entry.

### Step 11: Customise Workflows to Match Org SOP

This is a critical step. See Section 3 for the full guide. Do not skip this — importing the default templates without customisation means all users are working with generic approval thresholds that may not reflect the org's authority matrix.

### Step 12: Create Initial Users and Roles

In the browser, navigate to **Settings → User** and create user accounts for the organisation.

Fundara roles to assign (at minimum):
- **Fundara Administrator** — full access, usually for the Finance Manager or IT admin of the org
- **Finance Officer** — can create and approve financial transactions
- **Program Manager** — can create and manage Funds, Programs, Activities
- **Field Staff** — can create Cash Advance requests and liquidations
- **Auditor** — read-only access to financial and evidence records
- **Donor Viewer** — limited access to approved donor reports only

Always assign the most restrictive role that covers the user's actual job. Follow the principle of least privilege.

Set up **Two-Factor Authentication** for Finance Officer, Fundara Administrator, and any role with submit/approve authority on financial documents:

**Settings → My Settings → Two Factor Authentication**

Or enforce it system-wide:
**Setup → System Settings → Two Factor Authentication → Required**

### Step 13: Set Up Backup for the New Site

Every site must have its own backup schedule. Do not assume the server-wide backup covers per-site data correctly — verify it explicitly.

**Add the new site to the backup script** (or create one if it does not exist):

```bash
# Example: /home/frappe/scripts/backup-sites.sh
#!/bin/bash
BENCH=/home/frappe/frappe-bench
REMOTE=backup-server:/backups

cd $BENCH
bench --site ngoa.fundara.id backup --with-files
bench --site ngob.fundara.id backup --with-files
bench --site <orgname>.fundara.id backup --with-files  # ← add new site here

# Sync to remote
rsync -avz sites/ $REMOTE/
```

Schedule via cron:

```bash
crontab -e
# Add:
0 2 * * * /home/frappe/scripts/backup-sites.sh >> /var/log/fundara-backup.log 2>&1
```

Verify the first backup runs successfully:

```bash
bench --site <orgname>.fundara.id backup --with-files
ls sites/<orgname>.fundara.id/private/backups/ | tail -3
```

### Step 14: Onboarding Checklist for the Organisation

Hand this checklist to the organisation's Fundara Administrator before they invite their team:

- [ ] Administrator has logged in and changed the initial password
- [ ] System Settings reviewed: timezone, language, date format, currency
- [ ] Chart of Accounts reviewed and any org-specific accounts added
- [ ] Cost Centers created to match the org's departmental structure
- [ ] Organisation Profile completed
- [ ] Workflow configurations reviewed and customised (with Finance Domain Expert if Cash Advance or Purchase Invoice workflows were changed)
- [ ] All user accounts created and roles assigned
- [ ] Two-factor authentication enabled for Finance and Admin roles
- [ ] At least one Fund created (or the opening balance import completed)
- [ ] At least one Funding Source created
- [ ] Opening balances imported (if migrating from another system)
- [ ] One end-to-end transaction tested: Cash Receipt → Fund Balance update
- [ ] One end-to-end Cash Advance tested: Draft → Approved → Paid → Liquidated
- [ ] Backup verified — confirm admin can see backup files in the admin panel
- [ ] Organisation has acknowledged the data responsibility agreement

## 3. Workflow Configuration per Organisation

This section is the most operationally significant part of this guide. Read D-06 in `DECISIONS.md` before proceeding.

### The Fundamental Principle

The workflows shipped with Fundara are **default templates**. They represent a reasonable baseline SOP for a typical Indonesian NGO. They are not mandatory rules. Every organisation has its own approval SOP, its own authority matrix, and its own risk tolerance.

When onboarding a new organisation, the question is not "does this org accept the Fundara workflow?" The question is: **"what is this org's actual SOP, and how do we configure the Fundara workflow to match it?"**

### How to View and Edit Frappe Workflow for a Specific Site

Frappe Workflows are stored in the site's database, not in shared code. This means editing a workflow on `ngoa.fundara.id` does not affect `ngob.fundara.id`.

**To view existing workflows:**

Log in to the site → **Search Bar → Workflow** → open the workflow list.

Or navigate directly: `https://<orgname>.fundara.id/app/workflow`

**To edit a workflow:**

Open the workflow, make changes, and save. Changes take effect immediately — no restart required. However, documents that are already mid-workflow (in a non-Draft state) are not retroactively affected.

**Key workflow fields to understand:**

- **Document Type:** which DocType this workflow controls (e.g., `Advance`)
- **Workflow States:** the states a document can be in (Draft, Pending Approval, Approved, Paid, etc.)
- **Transitions:** which user roles can move a document from one state to another
- **State field:** the field in the DocType that stores the current state (do not change this)

### What to Discuss with the Organisation Before Customising

Before the first customisation session, gather this information from the org's Finance Manager or Programme Director:

**Authority Matrix:**

| Question | Example Answer |
|---|---|
| Who can approve a Cash Advance below Rp 5,000,000? | Finance Officer |
| Who can approve a Cash Advance above Rp 5,000,000? | Finance Manager + Programme Director |
| Is dual approval required above a certain amount? | Yes, above Rp 50,000,000 |
| Who has final approval authority for fund transfers? | Executive Director |
| Who can approve a Purchase Request? | Procurement Officer + Finance Manager |
| Is there a Board-level approval threshold? | Yes, above Rp 100,000,000 |

**Roles:**
- What roles does this org actually use? (Some orgs don't have a dedicated Procurement Officer)
- Do any roles overlap? (e.g., the Finance Manager also acts as the Procurement Officer)
- Who is the fallback approver if the primary approver is absent?

**Process quirks:**
- Are there document categories that bypass the standard approval? (e.g., emergency disbursements)
- Does the org require physical signatures in addition to system approval?
- Does the org's donor require specific documentation before approval can proceed?

### Which Workflows Are Safe to Customise

**Safe to customise without Finance Domain Expert review:**

| Workflow | What you can safely change |
|---|---|
| Cash Advance | Approval thresholds, number of approval steps, which roles can approve |
| Purchase Request | Approval thresholds, approval roles |
| Travel Request | Approval roles and thresholds |
| Fund Transfer (internal) | Approval roles |
| Field Report | Review and approval roles |

**Requires Finance Domain Expert review before customising:**

| Workflow | Why it requires expert review |
|---|---|
| Cash Advance (Paid state transition) | The Paid state triggers a GL entry (Payment Entry). Changing who can trigger this affects accounting. |
| Purchase Invoice (approval and posting) | Posting a Purchase Invoice creates GL entries. Any change to the approval-to-post flow affects the books. |
| Cash Receipt (submission) | Submission creates GL entries. |
| Liquidation (submission and acceptance) | Affects the outstanding advance clearing and expense posting. |
| Fund Transfer (cross-fund) | Affects fund balances and, if currencies differ, exchange rate accounting. |
| Journal Entry (submission) | Direct GL impact. |

The rule: if the workflow state transition triggers a GL entry creation or reversal, it must be reviewed by someone who understands the accounting implications.

### Template Customisation Guide

**Scenario: Changing an approval threshold**

The default Cash Advance workflow has a single approval step (Finance Officer approves all advances). An organisation wants: Finance Officer approves up to Rp 5,000,000; Finance Manager approves above.

1. Open the Cash Advance workflow on the org's site
2. The current transition is: `Draft → Pending Approval → Approved` (Finance Officer role)
3. Add a new state: `Pending Manager Approval`
4. Add a new transition rule:
   - From: `Pending Approval`, Condition: `doc.total_amount > 5000000`, Action: move to `Pending Manager Approval`
   - From: `Pending Manager Approval`, Allowed: `Finance Manager` role, Action: move to `Approved`
5. The existing transition (Finance Officer → Approved) adds a condition: `doc.total_amount <= 5000000`
6. Save the workflow

Test by creating a Cash Advance of Rp 3,000,000 (should reach Approved after Finance Officer approval) and another of Rp 10,000,000 (should stop at Pending Manager Approval until Finance Manager approves).

**Scenario: Adding a department head approval step**

An organisation requires the Programme Director to approve any advance tied to a programme:

1. Add a new state: `Pending Programme Director Approval`
2. Add a transition: from `Pending Approval`, if `doc.project != ""` (or `doc.program != ""`), move to `Pending Programme Director Approval`
3. Add a transition: `Programme Director` role can move from `Pending Programme Director Approval` to `Approved`
4. The Finance Officer transition applies only if `doc.project == ""`

**Scenario: Removing an approval step**

A small org with a single Finance Officer acting as both approver and payer wants to simplify the workflow — Draft goes directly to Approved on submission.

This is safe for the Cash Advance workflow, as long as the Finance Officer's role has the authority to both approve and trigger the Paid state. Make the change, document it in the org's onboarding notes, and flag it as a governance decision made by the org's leadership (not a Fundara default).

**Scenario: Disabling a workflow entirely**

Some orgs may not use certain features (e.g., they don't do Purchase Requests — they go straight to Purchase Invoices). In this case:

1. Open the workflow for the unused DocType
2. Set it to **Inactive** (or do not import it during onboarding)
3. Note this in the org's configuration record

### Caution: Workflows and Frappe's Is_Cancelled / Submit Logic

Frappe's submit/cancel mechanism is separate from the workflow state machine. A document can be in `Approved` workflow state but not yet submitted in Frappe's sense (i.e., `docstatus = 0`). The GL entry is created on `docstatus = 1` (submit), not on workflow state change, unless a server script or hook explicitly triggers posting on a specific workflow state.

Verify this behaviour for each DocType in Fundara before promising an org that their approval triggers the accounting entry. When in doubt, check `fundara/hooks.py` and the relevant DocType controller for `on_submit` behaviour.

## 4. Site Isolation Verification

### What Frappe Multi-site Guarantees

- **Database isolation:** Each site has its own MariaDB database. A query on `ngoa.fundara.id` cannot touch the `ngob_fundara_id` database. This is enforced at the Frappe connection layer — the site's database credentials are scoped to its own database only.
- **File isolation:** Private files are stored under `sites/<sitename>/private/` and are not served to other sites.
- **User isolation:** Users are stored per-site. A user account on `ngoa.fundara.id` does not exist on `ngob.fundara.id`.
- **Session isolation:** Sessions are scoped to the site via the `HTTP_HOST` header.

### What Frappe Multi-site Does NOT Guarantee

- **Worker isolation:** Background workers are shared. A long-running job on one site consumes worker capacity that affects all sites on the same server. A stuck or CPU-heavy job on one site can degrade responsiveness on other sites.
- **Cache isolation:** Redis cache is shared. In theory, Frappe namespaces cache keys per site, but a Redis misconfiguration or a Frappe bug could cause cross-site cache pollution. Monitor for this in production.
- **Resource isolation:** If one site runs a very large report generation or import, it can exhaust server RAM and affect other sites.
- **Code isolation:** All sites run the same version of `fundara`, `erpnext`, and `frappe`. A bug in a code update affects all sites simultaneously.

### Isolation Test Checklist

Run these tests after adding a new site and before handing it over to the organisation.

**Test 1: User cannot log in cross-site**

1. Create a user `testuser@ngoa.fundara.id` on `ngoa.fundara.id`
2. Attempt to log in to `ngob.fundara.id` with the same credentials
3. Expected result: login fails with "incorrect password or user does not exist"

**Test 2: Data does not bleed across sites**

1. Create a Fund named `Test Cross-Site Fund` on `ngoa.fundara.id`
2. Log in to `ngob.fundara.id`
3. Navigate to the Fund list
4. Expected result: `Test Cross-Site Fund` does not appear in `ngob.fundara.id`'s list

```bash
# Verify at the database level
mysql -u root -p
SHOW DATABASES LIKE '%fundara%';
USE ngoa_fundara_id;  # adjust to actual db name
SELECT name FROM tabFund WHERE name LIKE '%Test Cross-Site%';
USE ngob_fundara_id;
SELECT name FROM tabFund WHERE name LIKE '%Test Cross-Site%';
# Should return 0 rows in ngob
```

**Test 3: API does not allow cross-site access**

```bash
# Get a session cookie for ngoa
curl -c /tmp/ngoa-cookies.txt \
  -X POST https://ngoa.fundara.id/api/method/login \
  -d "usr=administrator&pwd=<admin-password>"

# Attempt to use that session cookie against ngob
curl -b /tmp/ngoa-cookies.txt \
  https://ngob.fundara.id/api/resource/Fund
# Expected: 403 Forbidden or 401 Unauthorized, NOT a list of ngob's funds
```

**Test 4: File attachments are isolated**

1. Upload a file to a Fund record on `ngoa.fundara.id`
2. Note the file path from the attachment URL
3. Attempt to access that file path on `ngob.fundara.id`
4. Expected result: 403 or 404 — the file is not accessible from the other site

**Test 5: Background job stays on its site**

1. Trigger a background job on `ngoa.fundara.id` (e.g., generate a large report)
2. Check the job queue on `ngob.fundara.id` — confirm the job does not appear in `ngob`'s job list

```bash
bench --site ngob.fundara.id console
# frappe.get_all("RQ Job", fields=["name", "status", "site"])
# All returned jobs should have site = "ngob.fundara.id"
exit()
```

## 5. Decommissioning a Site

When an organisation ends its use of Fundara, their site must be decommissioned in a controlled manner. Data must be preserved according to the data retention policy before deletion.

### Step 1: Confirm Decommissioning Intent

Before any action, get written confirmation from the organisation's authorised representative. Do not proceed based on verbal instruction. The confirmation should state:
- The site to be decommissioned
- The effective date
- The data export format required
- Whether the organisation wants a database export, or only specific report exports
- The data retention period required (minimum as per applicable law)

### Step 2: Export Organisation Data

**Full database export (recommended):**

```bash
bench --site <orgname>.fundara.id backup --with-files
ls -lh sites/<orgname>.fundara.id/private/backups/ | tail -5
```

This creates:
- A compressed SQL dump of the database
- A tar archive of private files (attachments, backups)
- A tar archive of public files

Package these for the organisation:

```bash
mkdir -p /export/<orgname>-export-$(date +%Y%m%d)
cp sites/<orgname>.fundara.id/private/backups/<latest-db-backup>.sql.gz \
   /export/<orgname>-export-$(date +%Y%m%d)/
cp sites/<orgname>.fundara.id/private/backups/<latest-private-files>.tar \
   /export/<orgname>-export-$(date +%Y%m%d)/
cp sites/<orgname>.fundara.id/private/backups/<latest-public-files>.tar \
   /export/<orgname>-export-$(date +%Y%m%d)/
```

**Report exports (supplementary):**

If the organisation requests specific reports in Excel or PDF format, generate and export them before decommissioning. This is separate from the database dump and provides human-readable records.

Typical reports to export before decommissioning:
- Fund Utilization Report (all time)
- Trial Balance (all fiscal years)
- Donor Financial Reports (all donors)
- Advance Aging Report
- ISAK 35 Financial Statements (all fiscal years)

### Step 3: Retain Backup for Required Period

Do not delete the backup immediately after export. Retain it on secure storage for the legally required period. For Indonesian organisations, financial records typically require a minimum 5-year retention. Confirm with the organisation's legal obligation.

Store the retained backup in encrypted offsite storage:

```bash
# Encrypt the export archive
gpg --symmetric --cipher-algo AES256 \
  /export/<orgname>-export-$(date +%Y%m%d).tar.gz

# Upload to remote storage
aws s3 cp /export/<orgname>-export-$(date +%Y%m%d).tar.gz.gpg \
  s3://fundara-archive/<orgname>/
```

Document the retention schedule and the storage location in a decommissioning record.

### Step 4: Disable User Access

Before deleting the site, disable user access so no data can be added or viewed while the decommissioning is in progress:

```bash
bench --site <orgname>.fundara.id set-maintenance-mode on
```

Or disable the site in Nginx by removing it from the site list and reloading Nginx.

### Step 5: Remove the Site

**This is irreversible. Ensure the backup is verified and stored remotely before this step.**

```bash
# Verify backup is on remote storage first
aws s3 ls s3://fundara-archive/<orgname>/

# Drop the site
bench drop-site <orgname>.fundara.id --db-root-password <mariadb-root-password>
```

This command:
- Drops the site's MariaDB database
- Removes `sites/<orgname>.fundara.id/` directory
- Removes the site from Frappe's site list

**Verify the database was dropped:**

```bash
mysql -u root -p -e "SHOW DATABASES;" | grep <orgname>
# Should return no results
```

### Step 6: Remove DNS and SSL

Remove the DNS A record for `<orgname>.fundara.id` at the DNS provider.

Remove the SSL certificate:

```bash
sudo certbot delete --cert-name <orgname>.fundara.id
```

### Step 7: Update Nginx Configuration

```bash
bench setup nginx
sudo nginx -t
sudo systemctl reload nginx
```

### Step 8: Document the Decommissioning

Record the following in your infrastructure log or runbook:

- Date of decommissioning
- Organisation name and site URL
- Date backup was created and verified
- Storage location of retained backup
- Retention expiry date
- Who authorised the decommissioning
- Who performed the decommissioning

## 6. Capacity Planning

### How Many Sites Can One Server Handle

Site capacity depends on concurrent active users per site, data volume, and the frequency of background jobs. Use the production profiles from `arsitektur.md` as a starting point:

**Profile B — Small NGO Server (4 vCPU, 8 GB RAM, 150 GB SSD)**

| Scenario | Estimated max sites |
|---|---|
| Sites with < 10 concurrent users, light background jobs | 3–5 sites |
| Sites with 10–30 concurrent users, regular reports | 1–2 sites |

**Profile C — Medium Server (App + DB separated, 8–16 GB RAM app, 16–32 GB RAM DB)**

| Scenario | Estimated max sites |
|---|---|
| Sites with < 10 concurrent users | 8–12 sites |
| Sites with 10–30 concurrent users | 4–6 sites |
| Mixed light and medium usage | 6–8 sites |

These are conservative estimates. The primary bottleneck for Frappe multi-site is usually:
1. RAM consumption by Gunicorn workers (each worker holds a Python process in memory)
2. MariaDB connections per site
3. Background job queue congestion when multiple sites run heavy reports simultaneously

### Monitoring Site-Level Resource Use

Add per-site monitoring to catch resource hogs early:

```bash
# Check database size per site
mysql -u root -p -e "
SELECT table_schema AS 'Site Database',
  ROUND(SUM(data_length + index_length) / 1024 / 1024, 1) AS 'Size (MB)'
FROM information_schema.tables
WHERE table_schema LIKE '%fundara%'
GROUP BY table_schema
ORDER BY SUM(data_length + index_length) DESC;
"
```

```bash
# Check active connections per site database
mysql -u root -p -e "SHOW PROCESSLIST;" | grep fundara
```

Watch for sites whose databases grow beyond 5 GB — large databases slow down migrations and backups disproportionately. If a site's database is large due to file attachments, consider moving file storage to S3-compatible object storage (see `arsitektur.md` section 18.5).

### When to Add Another Server

Consider adding a second server or moving to a Profile D (load-balanced, separate DB) setup when any of these conditions are met:

- Sustained CPU usage above 70% during business hours
- RAM pressure regularly causes swapping (check `free -h` and `vmstat`)
- Backup window overruns (backup takes so long that the next day's window starts before the previous finishes)
- Site count exceeds 10 with any active users
- Any single site requires dedicated capacity (high-volume donor campaigns, large-scale evidence imports)

### Load Balancing Options for High-Site-Count Scenarios

When the site count grows to 20+ or traffic demands exceed a single app server, the deployment architecture moves to Profile D from `arsitektur.md`:

```
[Load Balancer]
      ↓
[App Server 1] ─┐
[App Server 2] ─┼── [Redis]
[Worker Server] ┘
      ↓
[MariaDB Primary]
      ↓
[Replica / Backup]
      ↓
[S3-compatible File Storage]
```

**Key changes required for multi-app-server deployment:**

- File storage must move to S3-compatible object storage (local disk files cannot be shared across multiple app servers)
- Redis must be on a separate server accessible to all app servers
- Session management must use Redis (Frappe does this by default)
- MariaDB must be accessible from all app servers on the internal network
- Nginx load balancer must use sticky sessions or a session-aware load balancing strategy, since Frappe session state is stored in Redis (sticky sessions are simpler and safer for Frappe)

This architecture is significantly more complex to operate. Engage with the Frappe community documentation on horizontal scaling before attempting this configuration.
